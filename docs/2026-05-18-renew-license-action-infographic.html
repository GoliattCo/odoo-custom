Renew License — Atomic Action v2

Single mutation that revokes the old license + issues a new one in the same Drizzle transaction. Either both happen or neither does. Audit gets one license.renew row tying the pair via correlation.

GWhy an atomic action

Today renewal is: (a) Revoke the old license, (b) Mint a new one with the same customer_ref + image_sha256 + a fresh term. Two clicks, two audit rows, two opportunities to leave the customer inconsistent if (b) fails after (a) succeeded.

ΔBefore vs After

Today — Manual 2-step

Brittle

  • Click Revoke → reason → confirm → wait
  • Navigate to /licenses/new → fill form → submit
  • 2 unrelated audit rows; correlation only via timestamp proximity
  • If step 2 fails (DB blip, validation): customer locked out
  • Operator manually copies the new LICENSE_ID + emails customer
→
After — Atomic Renew

Transactional

  • Click Renew… on the detail page
  • One dialog: termDays (365), graceDays (14), optional reason
  • Confirm → single Drizzle transaction commits both ops
  • One license.renew audit row with {old_id, new_id, term_days, grace_days, reason} payload
  • Auto-redirect to NEW license's detail page

⚛Transaction Semantics

Implementation lives inside db.transaction(async tx => { ... }). Postgres rolls back automatically on any throw; the audit row + UPDATE + INSERT are all-or-nothing.

Transactional steps

  1. SELECT old license by id. Throw NOT_FOUND if missing, CONFLICT if already revoked_at IS NOT NULL.
  2. UPDATE enterprise_licenses SET revoked_at = NOW(), revoked_reason = 'renewed: superseded by ' || new_id
  3. INSERT enterprise_licenses — copies customer_ref, image_sha256, allowed_modules from old; new id; new expires_at + grace_until from term/grace inputs.
  4. INSERT audit_log with action='license.renew', payload tying both ids.
  5. Return { oldLicenseId, newLicenseId, newExpiresAt, newGraceUntil }.
✅ All 4 succeed → COMMIT. Customer sees new LICENSE_ID; addon's next hourly check picks up the change.
❌ Any throw → ROLLBACK. Old license stays unrevoked; no orphan new row. Operator retries.

?Open Questions

Copy notes from old to new?
No. Stale notes (e.g. "non-payment, 90 days past due") shouldn't carry to the renewal. New license starts with empty notes.
Change allowed_modules at renewal?
v1: copies old → new (no surprises). If a customer renews with different modules, operator uses the manual revoke+mint escape hatch.
Auto-email customer the new LICENSE_ID?
v1: manual (matches today). v2 nice-to-have: optional email checkbox in the dialog firing Resend.
Renew a license with future expires_at?
Allowed; operator decides. New license starts now, overlapping the old. Dialog description warns about the overlap.
