4The Four Surfaces

PR 1

Tenant CRUD

/tenants · /tenants/[id]
tenants.list tenants.get tenants.update tenants.markUntrusted

Highest-value, most-needed today. Filterable table (state: active / pending / suspended / archived), detail page with subscription + plan + backups + provisioning_jobs cross-references. Read-only ships first; mutations follow in PR 1.1.

Order: first (highest-value)
PR 2

Backup Catalog Viewer

/backups · /backups/[id]
tenantBackups.list tenantBackups.get tenantBackups.markUntrusted

Read-mostly. Filter by tenant / type (full / wal / filestore_tar) / date range / state. markUntrusted is the only mutation; mostly used during quarterly restore drills. Audit-logged.

Order: second — independent of tenant CRUD, useful in drills
PR 3

Plan CRUD

/plans · /plans/[code]
plans.list plans.create plans.update

Lowest churn — plans rarely change. Edit price_cop_cents / price_usd_cents / seat_included / module list / feature flags. Mid-cycle price changes need a confirm-by-typing-the-code guard (blast radius warning).

Order: third — low priority
PR 4

Manual Provision

/tenants/new
tenants.provision

Operator-driven tenant provisioning (vs the portal's signup flow). Highest blast radius — creates real infra on Railway/Fly. v1 = simple click-to-provision. v2 = N=2 approval gate via a "pending → approved → ran" state machine (separate spec).

Order: fourth — depends on tenant CRUD shipping first

⚠Blast-Radius Hazards

Provision

Creates real infra on Railway/Fly

Hard to undo (apps, volumes, secrets, costs). v2 spec: require N=2 operator approval via state machine. v1 keeps the manual CLI escape hatch.

Plan price edits

Affects existing subscriptions?

Mid-billing-cycle changes: retroactive vs new-signups-only? Document in UI before allowing save. Require typing the plan code to confirm.

markUntrusted

Doesn't actually delete

Just flips a flag the restore-drill respects. Real cleanup: a janitor cron with 30-day delete_at horizon (mirrors license-expiry-reminders pattern).

"drop tenant"

Intentionally absent

No UI button for destructive tenant teardown — CLI-only forever, with manual operator approval. Same rule as today.

?Open Questions

Provision approval gate now or later?
v1 = simple click-to-provision. v2 = N=2 gate when the team grows past one operator.
Tenant detail page scope?
Show recent license_checks, backups, provisioning_jobs inline (each ~10 entries with deep-link to /backups?tenant=…). Operator wants one-stop visibility.
Per-surface CODEOWNERS?
Tenant + Plan CRUD → security-leads review. Backup catalog → prod-deployers-only. Update .github/CODEOWNERS when each surface ships.
Plan changes retroactive?
Decide BEFORE letting operator save. UI must show "this change will affect N existing subscriptions" warning.
