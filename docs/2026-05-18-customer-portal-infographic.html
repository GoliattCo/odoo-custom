Customer Portal — License Self-Service v1

Enterprise self-host customers log in, see their own license status (active / grace / expired / revoked), download their pre-filled install bundle, and watch their license-check heartbeats land. Counterpart to the operator's /licenses dashboard.

GGoal & Non-Goals

Goal

Give enterprise self-host customers a self-service dashboard for their license: status, upcoming expiry, install bundle download, past renewals. Reduces operator handhold for post-onboarding "what's my LICENSE_ID?" / "when does this expire?" tickets, and gives the customer a place to verify their license is healthy from Goliatt's perspective without emailing support.

Non-Goals (deferred or operator-only)

  • Customer can mint / revoke / renew (operator-only)
  • Show LICENSE-SIGNING private key
  • Reveal per-customer HMAC secret in cleartext
  • Customer support chat / ticket portal
  • Multi-license-per-customer UI (defer until 1st such customer)
  • License-check telemetry beyond their own

AAudience Split: Customer vs Operator

Same Clerk org, different publicMetadata.role. Customer hits portal app; operator hits admin app. Cross-routing redirects gracefully.

Customer · apps/portal

What they see

  • Their own license card (active / grace / expired / revoked) with days-until-expiry badge
  • Download install bundle — docker-compose template pre-filled with their LICENSE_ID, LICENSE_AUTHORITY_URL, image digest
  • License-check audit feed (their addon's hourly heartbeats)
  • Past license history (revoked + restored entries)
  • Support contact addresses
read-only
Operator · apps/admin

What they still own

  • Mint new license (/licenses/new)
  • Revoke / Restore (action bar on detail)
  • Renew (when v12 spec ships — atomic action)
  • Audit log across all namespaces
  • Per-customer HMAC secret management

STenancy Isolation: 4-Layer Auth

Each customer can only see THEIR rows. Cross-customer leak is a security incident. Every read path pins customer_ref from the Clerk session into the WHERE clause; no procedure accepts customer_ref as an input.

1
Middleware (proxy.ts) — Clerk session required. Unauthenticated → redirect to /sign-in.
2
Layout requireCustomer() — server-side. Reads publicMetadata.role. Operators get redirected to admin app; non-roled users → /not-authorized. Cached per-request via React.cache().
3
tRPC customerProcedure — same role check + extracts customer_ref from session. Every query appends eq(enterpriseLicenses.customerRef, ctx.session.customerRef) automatically; bypass is impossible.
4
Defense-in-depth in server actions — repeat requireCustomer() at the top of every action. Catches a non-roled user who knows a server-action URL directly.

πAPI Surface — customerSelf tRPC

One new router; no new HTTP routes; no schema migration. The existing enterpriseLicenses.* operator procedures stay untouched.

customerSelf.myLicense()query · customer-only
Returns the most-recent active OR most-recent overall license bound to the authed customer's ref. Includes derived status + days-until-expiry.
customerSelf.myLicenseHistory({ limit?, cursor? })query · paginated
Past licenses including revoked + restored. Uses the shared paginationSchema helper.
customerSelf.myInstallBundle()query · file response
Pre-filled docker-compose + .env template + README zip. Does NOT include the HMAC secret (operator delivered out-of-band at onboarding).
customerSelf.myLicenseChecks({ limit?, cursor? })query · audit feed
Recent audit_log rows where action='license.check' AND target_id IN (this customer's licenses). Lets the customer see their install is calling home.

RRollout

Canary. Portal app already has signup/onboarding routes; this adds a new authenticated section. No public traffic until you tell a customer. Five PRs, each independently revertible.

PR 1
Add customerProcedure + customerSelf router (additive backend).
PR 2
Route group app/(customer)/ + layout with requireCustomer() + dashboard page.
PR 3
Install-bundle download endpoint + history + license-checks pages.
PR 4
Operator action: update Clerk users to have publicMetadata.customerRef (manual; one customer).

?Open Questions

Customer identity model
Stick with customer_ref free text (v1), or introduce customers table + UUID FK on enterprise_licenses? Latter is more correct but adds migration scope. v2 question.
Multi-license per customer
v1 assumes 1:1. When a customer needs 2 licenses (two Odoo installs), the dashboard needs a picker. Defer until that customer exists.
Bundle format
Single docker-compose.yml? Tar with compose + README? zip recommended (compose + .env.template + customer install README).
HMAC secret recovery
Customer loses their SAAS_PROVISIONING_SECRET — can they self-serve? Probably not (security-sensitive). Document "contact support".
Staleness alert to customer
Customer's install fails to call home for >X hours: auto-email them? Or only operator? Defer to first-paying-customer feedback.
